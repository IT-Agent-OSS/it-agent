# UIUX Checklist

## Purpose

`uiux` が見る観点を短く運用しやすくまとめたチェックです。

## Checklist

- 最初の3秒で何のツールか分かるか
- タイトルや概要説明より、主機能の操作面と結果面が主役になっているか
- 主操作が明確か
- 見た目が弱くて価値が落ちていないか
- CLI の場合、first-run example と expected output がすぐ見えるか
- Web の場合、what it does と expected result が above the fold にあるか
- Web の場合、complete sample と incomplete sample の両方を one-click で再現した方が価値や限界が伝わるなら、その導線があるか
- Web の場合、主操作と主要結果が同一画面内で追えるか
- 1画面に収まらない場合、縮小で押し込む前に画面分割やステップ切り替えで見せられるか
- 分類や比較ツールでは、decision rule や confidence が visible か
- recommendation / verdict / reviewer action が主出力のツールでは、その主出力が detail view で最も目立っているか
- 入力を編集して結果変化を見るツールでは、current result の要約が edit 中も見えているか
- re-run / re-score / re-triage の結果が変わらない場合でも、実行されたことを示す一時フィードバックがあるか
- 入力から結果まで迷わないか
- 動画で見せる見どころがあるか
- 動画化する場合、1280x720 など最終フレーム基準で上下左右の見切れがないか
- 録画用の縮小表示は最後の手段であり、まずUI構成で収める方針になっているか
- ブラウザ成果物では、静止画つなぎより実ブラウザ操作の録画を優先すべきか確認したか
- 日本語向け発信なら、UI と補足説明が日本語で理解できるか
- `prefers-color-scheme: dark` に対応している場合、OS/ブラウザのダークモードを実際に強制して select や button などのコントロールが読めるか確認したか(CSSにdark blockがあること自体は根拠にしない)
- 標準化できるUI/UX学習があるか
- 過去の見た目テンプレートをなぞっているだけになっていないか
- 再利用してよいのは原則であり、レイアウトや演出は今回のテーマに合わせて再判断しているか
- サンプル/プリセットボタンがある場合、そのボタン 1 回でデモが完結するか（入力セット + 自動実行）、または次アクションへの明示的な誘導（ラベル変化・矢印・ハイライト）があるか
- 主操作ボタンが空入力・無効状態で実行された場合、テキスト・ボーダー変化・一時ラベル変更のいずれかで視覚的フィードバックを返すか（silent return は UI として成立しない）
- 同一入力に対して再実行した場合でも、「実行された」ことが分かる一時フィードバック（ボタン disabled + ラベル変化、または完了トースト）があるか
- 凡例・legend が結果表示後もスクロールなしで参照できる位置にあるか、または結果が出る前に一度は目に入る位置に置かれているか
- スコア・グレード・ランクを出力するツールでは、その判定基準（閾値・比率・スケール）が Grade/Score 要素の tooltip または legend に常時表示されているか（「Grade D」のような結果だけでは判断基準が伝わらない）
- キーボードショートカットのヒントは、そのショートカットが適用される操作ボタンの直後・または inline ラベルとして配置されているか（`margin-left: auto` による右端配置は操作ボタンとの関連が切れるため不可）
